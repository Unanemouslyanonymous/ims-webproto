⚠️ The Problem

Static kernel schedulers — SLURM in HPC clusters, the Kubernetes default scheduler in cloud environments — were designed for an era of homogeneous hardware and predictable workloads. They lack the feedback loops necessary to respond to live hardware signals.

⛔ Silent Failure Modes in Production HPC

GPU XID errors accumulate silently until throughput collapses. NVMe wear crosses the 85% cliff and write amplification explodes. NUMA topology mismatches cause 30–40% memory bandwidth loss on multi-socket systems. Rack PDU budget overruns trigger emergency throttles at the BMC level, stalling entire job queues — with zero visibility from the scheduler.

No existing scheduling tool learns from these hardware signals autonomously. Operators rely on manual runbooks, reactive alerts, and post-mortem analysis. The cost is measured in CPU-hours wasted, GPU clusters underutilised, and SLA breaches that compound silently across multi-week HPC campaigns.

Root Causes of Scheduler Blindness
  • No hardware feedback loop — SLURM and k8s schedulers read node metadata at job submission, not real-time hardware telemetry.
  • No online learning — Affinity tables and priority weights are hand-coded and require redeployment to update.
  • No cross-signal correlation — Power, thermal, GPU health, and storage wear signals exist in separate monitoring silos with no unified dispatch logic.
  • No security-aware placement — Standard schedulers have no concept of hardware attestation or TEE placement requirements at the job level.

⚡ The Product

IMS (Intelligent Meta Scheduler) is a meta-scheduler middleware layer that sits between the OS kernel and the existing job dispatcher. It operates as a transparent proxy — consuming hardware telemetry, computing optimal placement decisions via online reinforcement learning, and either advising or directly actuating the underlying scheduler.

IMS observes real hardware telemetry directly from the hardware abstraction layer: RAPL MSR registers for per-socket and per-core power, NVML for GPU health scores and error counts, SMART attributes for NVMe wear percentage and write amplification, CPU MSR P-state registers for frequency scaling state, and IPMI/Redfish endpoints for rack-level power budget and thermal zone data.

✅ Key Architectural Principle

IMS requires no kernel patches. It instruments hardware via existing OS interfaces (perf_event, sysfs, NVML library, IPMI userspace tools) and attaches to the scheduler via a lightweight gRPC dispatch advisory protocol. Deployment takes minutes, not maintenance windows.

RL Scheduling Model

The core scheduler is a tabular Q-learning agent with state factored over hardware health vectors. The reward function is a weighted composition of: throughput per watt (primary), job completion latency (secondary), security constraint satisfaction (hard constraint, zero violation tolerance), and hardware wear minimisation (regulariser term).

Affinity is computed algebraically from the resource demand vector of each process type against the current capability vector of each schedulable unit — no hand-coded affinity tables, no manual tuning. The agent adapts to live hardware events (GPU degradation, NVMe wear cliff, thermal emergency) without redeployment, via the exploration-exploitation balance governed by the decaying epsilon parameter.

⚡ Physics-Aware RL — The Core Moat

IMS computes job-to-unit affinity from the resource demand vector of the process (power draw, memory bandwidth, I/O intensity, security class) against the real-time capability vector of each schedulable unit. This is physics-derived, not table-driven. The agent learns the affinity function from hardware feedback — it does not need to be told that a GPU with 12 XID errors in the last hour should be deprioritised for FP32 matrix workloads. It discovers this from the reward signal.

The model weights are transferable across clusters of similar hardware generation, enabling a federated learning flywheel: the more clusters run IMS, the better the shared base model, creating a data network effect that widens the moat over time.

National AI Telemetry Standard for HPC — Version 1.0 (Draft)
Recommendation of standardisation principles for AI telemetry and self-healing frameworks within the national OS ecosystem.
Status: Draft for Public Comment  ·  Classification: Unclassified

📋 Rationale

Current HPC deployments instrument their hardware using a patchwork of vendor-specific APIs, custom Prometheus exporters, and site-local schemas. This fragmentation has three critical consequences for AI-driven infrastructure management:

  • Model non-transferability — RL models trained at one national lab cannot be applied at another because signal schemas, sampling rates, and retention policies are incompatible.
  • Federated training barrier — Federated learning across multiple sites requires a common data representation. Without a standard, each federation attempt requires costly custom ETL pipelines.
  • Audit and compliance gaps — Fragmented telemetry makes it impossible to produce standardised audit evidence for supply chain, security, or regulatory purposes.

NATS-HPC 1.0 defines a minimal standard telemetry schema and self-healing trigger taxonomy to enable cross-site RL model transfer, federated training, and interoperable audit trails across participating national labs and sovereign AI infrastructure.

📡 Telemetry Data Model

All compliant IMS deployments must emit the following mandatory telemetry signals at the specified minimum sampling rate. Additional signals are permitted; removal of mandatory signals breaks NATS-HPC 1.0 compliance.

📊 OpenTelemetry Integration

IMS exports all telemetry signals in OTLP (OpenTelemetry Protocol) format. This makes IMS-instrumented clusters compatible with any OTLP-capable observability backend: Grafana Tempo, Jaeger, Honeycomb, Datadog, or self-hosted OpenTelemetry Collector pipelines.

📦 OTLP Export Specification

Metrics — Gauge instruments for all continuous signals (power, temperature, utilisation). Counter instruments for event counts (XID errors, dispatch events). Histogram for latency distributions (dispatch decision time, job completion time).

Traces — Every dispatch decision is wrapped in an OTLP trace span. Span attributes include: process type, chosen action, Q-values for all actions, hardware state vector at decision time, reward at completion (added retroactively via span update).

Logs — Structured JSON logs with reward breakdown per job completion. Compliant with OpenTelemetry Log Data Model. Emitted to OTLP log exporter or stdout (for capture by cluster log aggregator).

For sites that cannot expose OTLP endpoints externally, IMS supports a local Prometheus scrape endpoint at /metrics (port 9090 by default) as a fallback. The OpenTelemetry Collector can bridge between Prometheus scrape and OTLP push models.

💾 Retention Policy

30d Hot Tier NVMe SSD · full resolution
90d Warm Tier Object store · 10x downsampled
1y Cold Tier Tape / Glacier · event log only

Hot tier data is retained at full sampling resolution. Warm tier applies a 10× downsampling to continuous signals but retains all event-based records (dispatch events, self-healing triggers, reward signals) at full fidelity. Cold tier retains event-based audit records only — continuous telemetry is not retained beyond the warm tier window. All transitions are governed by IMS Retention Manager and are configurable per-site within the bounds defined above.

Security Hardening Protocol for Meta-Scheduler Layer — Version 1.0
Establishment of security hardening protocols for the meta-scheduler layer to ensure secure multi-tenant operations in high-density compute environments.
Classification: Controlled Unclassified Information (CUI) — Distribution: Authorised Personnel Only

🎯 Threat Model

The IMS threat model is scoped to the meta-scheduler layer as a high-value target in multi-tenant HPC environments. The following threat categories are considered in scope:

Tenant Isolation Breach
A tenant's secure workload is scheduled on a compute unit shared with or adjacent to an untrusted co-resident job. Includes both CPU and GPU co-location threats.
Severity: Critical
Model Poisoning
An adversary injects malicious reward signals or manipulates telemetry to bias the RL model toward insecure or degraded placement decisions over time.
Severity: Critical
Supply Chain Compromise
The IMS scheduler binary, dependencies, or container image is compromised pre-deployment via a compromised build pipeline or upstream dependency.
Severity: High
Side-Channel Attack
A co-resident job infers sensitive data from a secure enclave job via shared hardware resources: shared LLC cache timing, DRAM row-hammer, or power consumption patterns observable via RAPL.
Severity: High

Out-of-scope threats for SHP-MSL 1.0: network-layer attacks (handled by the cluster network security policy), physical access attacks (handled by data centre physical security controls), and application-layer vulnerabilities within tenant workloads themselves.

🛡️ Zero Trust Architecture

IMS implements Zero Trust principles as defined in NIST SP 800-207. The scheduler is not an implicit trust boundary — every dispatch decision is treated as a security event and requires attestation evidence before execution.

📜 Zero Trust Dispatch Invariants
FOR EVERY dispatch decision:
  1. Workload security class MUST be verified from job manifest (signed)
  2. Target unit TEE availability MUST be confirmed via hardware attestation
  3. Cross-tenant visibility on target unit MUST be zero (verified by isolation policy engine)
  4. Dispatch event MUST be signed with HSM key and appended to immutable audit log
  5. Reward signal computation MUST occur within TEE (no plaintext reward leakage)

Secure workloads (security class ≥ 2) require hardware-attested TEE placement. The IMS scheduler queries the attestation service before issuing a placement decision for any secure-class job. Unattested placement attempts are blocked, logged as security violations, and trigger an immediate operator alert.

Attestation reports are cached for a maximum of 60 seconds. Cached reports are invalidated immediately on receipt of any security-relevant hardware event (XID error, ECC spike, unexpected power state change). For air-gapped deployments, a local attestation verification service is provided as part of the IMS Enterprise package.

📦 SLSA Supply Chain Controls

✅ SLSA Level 3 Compliance

The IMS scheduler binary is built at SLSA Level 3. This means: the build environment is isolated and non-persistent; the build process is defined as code in a signed configuration file; provenance is generated by the build system (not the developer); and the provenance is cryptographically signed and verifiable by any party with access to the public key.

  • Signed SBOM — A Software Bill of Materials in SPDX 2.3 format is attached to every release, signed with the IMS release key (Sigstore Cosign).
  • Reproducible builds — IMS uses Nix as the build system. Any party with the Nix flake can reproduce the exact binary from source, bit-for-bit.
  • Policy-as-code — Scheduler security policies are expressed in OPA (Open Policy Agent) Rego. Policies are versioned, tested, and reviewed via the same PR process as source code.
  • Dependency pinning — All third-party dependencies are pinned by content hash (not semver range). Dependency updates require explicit approval via policy review.
  • Container image signing — OCI container images are signed with Cosign and verified at deploy time by the cluster admission controller.

📋 Multi-Tenant Isolation Rules

  1. Row 0 (the enclave row) is a reserved hardware partition dedicated to secure processes only. No general-purpose, low-security, or untrusted workload may be dispatched to Row 0 under any circumstance, including queue overflow conditions.
  2. Every job-to-unit mapping is logged with a cryptographic proof of placement: the dispatch event record includes the job ID, target unit ID, timestamp, attestation report hash, and the HSM-signed authorisation token.
  3. The reward signal for RL training is computed inside a Trusted Execution Environment (TEE). Raw reward components are never exposed in plaintext to any process outside the TEE boundary.
  4. RL model weights are stored in an encrypted volume (AES-256-GCM). Encryption keys are managed by an HSM (Hardware Security Module) and are never written to disk in plaintext. Key access requires two-party authorisation for Enterprise deployments.
  5. Any detected violation — a secure-class job dispatched to a non-enclave unit, an attested unit failing re-attestation, or a placement event without a valid HSM signature — triggers: (a) immediate job termination, (b) unit quarantine, (c) P1 operator alert with full evidence bundle, (d) automatic entry in the SIEM event stream.

📝 Audit Trail

Every dispatch event emitted by IMS is signed with the cluster's HSM key and appended to an immutable, append-only audit log. The log is WORM-compliant (Write Once, Read Many) and cannot be modified or deleted by any user, including cluster administrators, for the duration of the retention period.

Audit Event Fields (per dispatch)

  • Event timestamp (nanosecond precision, NTP-synchronised)
  • Job ID + tenant ID (anonymised in shared-log mode)
  • Target unit ID + attestation report hash
  • Security class of dispatched job
  • RL action taken + Q-values for all alternatives
  • Hardware state vector snapshot at dispatch time
  • HSM signature over the above fields (Ed25519)

Audit Export & SIEM Integration

  • Syslog (RFC 5424) export for legacy SIEM compatibility
  • CEF (Common Event Format) for Splunk / ArcSight
  • OCSF (Open Cybersecurity Schema Framework) for modern SIEM
  • Direct Kafka topic publish for stream-processing pipelines
  • Tamper-evidence: SHA-3 hash chain across sequential log entries
  • Quarterly log integrity attestation report available on demand

💡 Explainability Requirements

To satisfy Article 13 (EU AI Act) transparency obligations and the NIST AI RMF MEASURE function, every RL dispatch decision must emit the following structured fields:

  1. Process type — The workload class of the dispatched job (e.g., GPU_INTENSIVE, MEMORY_INTENSIVE, IO_BOUND, SECURE_ENCLAVE). Derived from the process type registry.
  2. Resource demand vector — The multi-dimensional resource requirement of the job at dispatch time: power budget, memory bandwidth, I/O intensity, security class, estimated runtime.
  3. Q-values for all actions — The full Q-value vector across all available schedulable units at decision time, enabling retrospective analysis of why a specific unit was selected over alternatives.
  4. Chosen action and rationale — The selected unit, whether the choice was greedy (exploitation) or random (exploration), and the primary hardware signal that differentiated the selected unit.
  5. Reward components at completion — The decomposed reward signal: throughput component, latency penalty, security constraint satisfaction score, hardware wear cost. Enables per-component accountability for model decisions.

🧬 Automated Diagnostics

IMS operates a self-monitoring diagnostics loop that continuously evaluates the health of the RL agent itself — not just the hardware it manages:

Reward EMA Trend Monitor
The exponential moving average of the reward signal is monitored for sustained decline. A declining EMA slope (< −0.05 over 500 steps) indicates an unresolved hardware event or a policy that has diverged from the optimal. Triggers: epsilon boost (force re-exploration) and operator advisory alert.
Policy Drift Detector
The distribution of Q-values across the action space is monitored via KL divergence from a reference checkpoint. Significant divergence (KL > 0.3) indicates the policy has shifted substantially — either due to hardware environment change (expected) or adversarial reward manipulation (unexpected). Triggers MEASURE alert and optional automatic rollback to last known-good checkpoint.
Telemetry Anomaly Detection
An isolation-forest anomaly detector runs over the incoming telemetry stream. Anomalous signal values (e.g., power draw suddenly reporting 0W on an active socket, GPU health jumping from 0.3 to 0.95 instantaneously) are flagged as potential telemetry tampering events, triggering audit log entries and telemetry quarantine.
Security Constraint Satisfaction Monitor
The security violation rate (secure jobs dispatched to non-enclave units) is monitored in real time. Any non-zero rate triggers immediate P1 escalation per SHP-MSL 1.0 Section 5. This monitor cannot be disabled or overridden by operator configuration.
